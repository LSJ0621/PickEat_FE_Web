## 회고 1: Playwright 병렬 경합 — bug-report 3건 실패 (2026-04-12)

### 증상

Playwright E2E를 기본 설정(`workers: 3`)으로 실행했다.

- 22건 중 17 passed / 3 failed / 2 flaky
- 실패 3건은 모두 `bug-report.spec.ts`
- 주 에러:
  ```
  page.goto: net::ERR_ABORTED; maybe frame was detached?
  ```
- 실패 스크린샷: `/bug-report`로 이동을 시도했으나 **로그인 페이지로 리다이렉트되어 있음**

### 근본 원인 3단계 분석

**1단계: 직접 원인**
`bug-report.spec.ts`가 `authedPage` fixture를 통해 `test@example.com`으로 로그인한다. 동시에 `auth.spec.ts`는 같은 `test@example.com`으로 로그인/로그아웃/회원가입 시나리오를 돌리고 있다. 두 spec이 같은 워커 풀에서 병렬 실행되며 **같은 BE 세션 상태를 경합**한다.

**2단계: 구조 원인**

- **시드 유저가 1명뿐이다.** `test@example.com` 단일 계정을 모든 E2E가 공유한다.
- Playwright 기본 `workers: 3` — 3개 spec이 동일 계정으로 동시에 로그인/로그아웃을 시도한다.
- BE의 refresh token rotation + Redis session 저장 로직은 단일 사용자의 동시 요청을 **레이스 컨디션 없이 처리하도록 설계되어 있지 않다** (현실적 요구사항에 없었음).
- 결과: `auth.spec.ts`의 로그아웃이 `bug-report.spec.ts`의 세션을 무효화 → 리다이렉트 → 네비게이션 abort.

**3단계: 시스템적 해결 선택지**

| 방안 | 장점 | 단점 |
|------|------|------|
| A. 워커별 시드 유저 분리 (`test-1@`, `test-2@`, `test-3@`) | 병렬 속도 유지 | 시드 인프라 복잡도 ↑, 워커 수 변경 시 시드도 동기화 필요 |
| B. 각 spec이 자체 유저를 생성 (E2E 전용 API) | 완전 격리 | BE에 "E2E 유저 생성" 엔드포인트 + 정책 필요 |
| **C. `workers: 1` (채택)** | 단순, 인프라 변경 0 | 속도 희생 |

**채택 근거**
현재 E2E 22건 × 2브라우저 = 44건, 총 실행 시간 약 1.4분. 파이프라인 예산으로 충분히 수용 가능. 복잡한 격리 인프라를 도입하기 전 **가장 단순한 해결책**을 택했다. 규모가 10배가 되면 방안 A/B를 재검토한다.

### 재발 방지

- `playwright.config.ts`에 `workers: 1`을 명시하고, 주석으로 **"왜 순차 실행인가"**를 한 줄 기록한다.
- 본 회고 문서에 결정 근거를 박제한다. 후속 개발자가 "왜 병렬을 안 켰지?"라고 물었을 때 이 문서를 참조할 수 있게 한다.

### 교훈

**병렬 테스트는 '공유 리소스'를 회피할 수 있을 때만 안전하다.** E2E에서 공유 리소스(DB 시드 유저, Redis 세션, 외부 API quota)가 존재하면 **격리 전략이 병렬화보다 선행되어야 한다**. 격리 비용이 속도 이득보다 크다면 순차 실행이 정답이다. 속도는 정량적이지만, 격리 실패로 인한 flaky test는 신뢰를 잃는 정성적 손실이다.

## 회고 2: 전략 §4.1 "내부 훅 의존성 원칙" 추가 (2026-04-12)

### 배경

공통 인프라 훅 테스트를 작성하면서 `useConfirmModal` 시나리오에 도달했을 때, 내부에서 다른 훅(`useModalScrollLock`, `useEscapeKey`)을 호출하는 컴포지션 패턴을 어떻게 검증할지가 애매했다.

기존 전략 문서(`docs/testing/test-strategy.md` §6.1 "행동 테스트, 구현 검증 금지")는 **이 경계 사례를 명시적으로 다루지 않았다**. 처음에는 "내부 훅도 의존성이니 호출 여부를 검증해야 한다"고 해석해 다음과 같이 작성했다:

```typescript
// 초기 시나리오 (문제 있음)
it('useModalScrollLock을 호출해야 한다', () => {
  renderHook(() => useConfirmModal());
  expect(mockUseModalScrollLock).toHaveBeenCalled();
});

it('useEscapeKey에 close 핸들러를 전달해야 한다', () => {
  const { result } = renderHook(() => useConfirmModal());
  expect(mockUseEscapeKey).toHaveBeenCalledWith(result.current.handleClose);
});
```

### 구현 직후 재검토에서 드러난 문제

테스트를 통과까지 시킨 뒤 다시 읽어보니, 이 접근은 §6.1 "구현 검증 금지"와 본질적으로 동일한 문제를 안고 있었다.

호출 검증은 **무엇을 렌더링/반환하느냐가 아니라 어떻게 구현하느냐를 고정시킨다**. 내부 훅을 동등한 다른 구현(예: 커스텀 event listener, context 기반 구현)으로 교체해도 외부 행동이 같다면 테스트는 통과해야 한다. 그러나 위 시나리오는 그 교체를 불가능하게 만든다. §6.1의 원칙이 "구현 세부사항에 의존하지 말 것"인데, 위 테스트는 **"어떤 훅을 호출했는가"** 라는 가장 구현 의존적인 사실을 고정시키고 있었다.

올바른 검증은 다음과 같아야 한다:

```typescript
// 좋음: 훅이 반환하는 상태/함수만 검증
const { result } = renderHook(() => useConfirmModal());
expect(result.current.showConfirmCard).toBe(false);

act(() => result.current.openConfirm());
expect(result.current.showConfirmCard).toBe(true);

act(() => result.current.closeConfirm());
expect(result.current.showConfirmCard).toBe(false);
```

`useModalScrollLock`, `useEscapeKey`의 **호출 여부나 인자**는 테스트 대상이 아니다. 그들이 제공하는 효과(모달 열릴 때 body 스크롤 잠김, Escape 시 닫힘)는 각각의 훅 테스트에서 보장한다.

### 수정 과정

1. 문제가 `useConfirmModal` 하나에 그치지 않는다고 판단했다. 컴포지션 훅은 이후에도 계속 등장할 것이므로, 전략 문서 자체에 명시적 규칙이 필요했다.
2. `docs/testing/test-strategy.md`에 §4.1 "내부 훅 의존성 원칙"을 신설:
   > Hook이 다른 Hook을 내부에서 호출하는 경우, 그 의존성 훅의 **호출 여부나 인자는 검증하지 않는다**. 테스트는 훅이 반환하는 상태/함수의 행동만을 대상으로 한다. 의존성 훅이 제공하는 효과는 해당 훅 자체의 테스트에서 보장한다.
3. 새 원칙에 맞춰 `useConfirmModal` 시나리오를 재작성하고, **다른 훅 테스트에도 같은 문제가 있는지 전수 재검토**했다. 이미 작성된 테스트 중 일부에서 유사한 "호출 검증" 패턴이 더 발견되어 함께 정리했다.

### 교훈

**전략 문서는 완벽할 수 없다.** 설계 시점에는 모든 경계 사례를 예측할 수 없고, 현실에서 부딪히는 순간이 곧 문서가 성장할 기회다. 이때 내가 지킨 원칙은:

1. **구체적 상황을 부록에 예시로 박제한다** — 추상 원칙만 적으면 다음에 같은 케이스를 만났을 때 또 헷갈린다.
2. **구현 직후에 바로 전략을 뜯어고치지 않는다** — 한 박자 쉬고 재검토한 뒤 반영한다. 즉흥 판단을 피하기 위함이다.
3. **전략 수정 후 기존 시나리오를 전수 재검증한다** — 새 원칙이 과거 작업에 영향을 주는지 반드시 확인한다. 이번에도 실제로 추가 수정이 있었다.

전략 문서는 돌에 새긴 법이 아니라 **살아있는 계약**으로 다뤄야 한다. 현실에서 경계 사례를 만나면 문서를 먼저 고치고, 이후 작업은 고친 원칙 위에서 이어간다. 이 한 번의 수정이 이후 모든 훅 테스트의 품질을 끌어올렸다.
